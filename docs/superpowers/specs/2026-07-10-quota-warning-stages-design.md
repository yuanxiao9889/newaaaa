# 余额预警单位与分阶段通知设计

## 背景

当前个人通知设置直接展示和编辑内部额度值，用户看到的数值与站点配置的金额单位不一致。例如站点使用人民币展示时，用户仍需填写类似 `500000` 的内部额度，难以理解。余额低于阈值后，每次消费还可能重复触发通知，即使存在小时级通知限流，仍会持续发送较多邮件。

## 目标

- 预警阈值输入使用站点当前配置的展示单位，例如 USD、CNY、自定义货币或 Tokens。
- 后端继续以内部 quota 单位保存阈值，保持现有接口、历史设置及计费逻辑兼容。
- 每轮低余额周期最多发送三次预警，分别对应配置阈值的 100%、50% 和 20%。
- 充值使余额恢复到配置阈值及以上后，重置阶段状态，允许下一轮预警。
- 邮件和其他通知中的余额使用站点配置单位，并展示扣费后的实际剩余额度。

## 前端设计

个人资料的通知设置仍使用现有 `quota_warning_threshold` API 字段，但表单内部区分展示值和保存值：

1. 加载用户设置时，通过现有 `quotaUnitsToDollars` 将内部 quota 转为站点展示值。
2. 标签和输入框附带当前单位。货币模式显示 USD、CNY 或自定义货币符号；Tokens 模式显示 Tokens。
3. 保存时通过现有 `parseQuotaFromDollars` 将展示值转换回内部 quota。
4. 不修改后端请求字段名称与已保存数据格式。
5. 提示文案说明会在阈值的 100%、50% 和 20% 三个阶段通知。

## 后端阶段设计

根据扣费后的实际剩余额度计算当前阶段：

- `remaining >= threshold`：正常状态，清除已发送阶段。
- `threshold * 50% <= remaining < threshold`：阶段 1（100% 阈值预警）。
- `threshold * 20% <= remaining < threshold * 50%`：阶段 2（50% 阈值预警）。
- `remaining < threshold * 20%`：阶段 3（20% 阈值预警）。

同一个低余额周期内，每个阶段只允许发送一次。如果单次消费跨过多个阶段，只发送当前最紧急阶段的一封，并将该阶段记为已发送，避免连续发送多封。

钱包余额使用扣费前余额减去本次实际消费得到扣费后余额。订阅额度使用订阅总额度减去最终已使用额度得到扣费后余额。通知正文统一使用该扣费后余额，不再使用旧的扣费前快照。

## 去重状态

采用与现有通知限流一致的 Redis + 内存降级方案：

- 状态键包含用户 ID、计费来源和来源 ID，钱包与不同订阅互不影响。
- Redis 可用时使用原子状态更新，阻止并发请求重复领取同一阶段。
- Redis 未启用时使用进程内并发安全存储。
- 余额恢复到阈值及以上时删除状态。
- 阈值发生变化时按新阈值重新计算并重置旧阶段状态。
- 状态记录当前阈值和已发送阶段，阈值变化时以新阈值开始一轮状态。
- Redis 状态保存 30 天并在每次阶段判断时续期，避免长期遗留；过期后低余额用户至多重新收到当前阶段的一封，不会瞬间补发三封。

现有 `NotifyUser` 通用小时级限流继续保留，作为第二层保护。

## 通知内容

通知标题保持现有余额或订阅额度即将用尽的语义。正文增加当前预警阶段，例如“余额已低于预警阈值的 50%”，并使用 `logger.FormatQuota` 按站点配置格式化实际剩余额度。Email/Webhook 保留充值链接，Bark/Gotify 使用短文本。

## 测试

后端增加确定性表格测试，覆盖：

- 阈值边界的阶段计算。
- 同一阶段不重复领取。
- 从阶段 1 进入阶段 2、再进入阶段 3 时各领取一次。
- 单次直接跌入阶段 3 时只领取阶段 3。
- 余额恢复到阈值后状态重置。
- 钱包和不同订阅的状态隔离。
- 通知使用扣费后余额。

前端通过类型检查、lint、格式检查及生产构建验证转换逻辑和组件改动。现有额度转换函数已被多个金额输入组件使用，本次复用而不新增重复转换实现。

## 非目标

- 不改变全局通知方式选择。
- 不改变通用通知限流环境变量。
- 不改变数据库中的 quota 存储单位。
- 不新增数据库表或迁移。
